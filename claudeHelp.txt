The mapping step (the only conceptually new p

This is what separates a "real" RAG evaluatio gives you (query, doc_id, section_id). Yourchunker turns each PDF into N chunks. Question: which of those N chunks should count as "gold" for that query?

The algorithm (industry-standard, called passage overlap matching):

For each query in the dataset:
  gold_section_text = corpus[doc_id]["section

  Build word n-gram set from gold_section_tex

  For each chunk in chroma_db where metadata.
    chunk_grams = 8-grams of chunk text
    overlap = |chunk_grams ∩ gold_grams| / |c
    if overlap >= 0.3:           # at least 30% of chunk is in gold section
      mark as gold

Why n-gram overlap and not exact substring? Puirks; whitespace-tolerant n-gram overlapsurvives those. Also handles the case where one chunk spans the gold-section boundary — it'll partially match.

Why ≥30% (not 50% or 80%)? A chunk that contains 30% of its tokens in the gold section is almost certainly about that
section. Higher thresholds miss boundary chune positives. 30% is a defensible default —record it in your eval config so future-you knows.

Edge case to handle: if a query's (doc_id, section_id) yields ZERO gold chunks (no chunk crosses the 30% threshold), drop
 that query. Otherwise it'll show up as Hit@1retable noise. Log how many you dropped.

2. What "gold" means

"Gold" is information-retrieval jargon for ground truth — the answer a human (or trusted source) marked as correct. The
word comes from search benchmarks like TREC ad documents and label "yes, this document isrelevant to this query." Those labels are the gold standard against which a retriever is measured.

In your eval pipeline, look at eval/metrics.py:14:
def hit_at_1(retrieved_ids, gold_ids):
    return 1 if retrieved_ids[0] in gold_ids else 0
This asks: "Did the retriever's #1 result matcorrect?" If yes → 1 point. If no → 0 points.Sum across all queries, divide by count = Hit@1 score.

Without gold_ids, this entire metric is impossible. You'd only have the LLM-judge metrics (faithfulness, correctness),
which conflate retrieval errors with generati if a bad answer was caused by retrieving thewrong chunk or by the generator screwing up correct context. That's why establishing gold is non-negotiable.

In synthesize.py we got gold_chunk_ids for free, because the synthesizer literally generated each question from a
specific chunk — that chunk's ID was, by consal benchmark you don't get this for free; youhave to derive it.

4. n-grams and why they're the right tool her

What an n-gram is: a sliding window of N cons

Example with "machine learning is a subset of
- 1-grams: ["machine", "learning", "is", "a", "subset", "of", "AI"]
- 3-grams: ["machine learning is", "learning et of", "subset of AI"]
- 8-grams: none (only 7 words). In your real chunks (~500 words) you'd get ~493 distinct 8-grams.

The mapping problem restated: given the text of a gold section (call it S) and the text of a candidate chunk (call it C),
 decide if C "is about the same content as" S

You could try simpler approaches first. Here'

Naive option A — exact substring match: "is C
Fails because PDFs are messy. Section text was extracted by the dataset authors using one parser; your chunks were
extracted by PyPDF2. The same paragraph may d
- Whitespace (single space vs double space vs newline)
- Unicode (em-dash vs hyphen, ligatures like
- Hyphenated line breaks ("re-\nsult" vs "result")

One character mismatch breaks a substring match across an entire chunk. Brittle.

Naive option B — word set overlap (Jaccard on unigrams):
Treat both S and C as bags of words; compute

Fails for two reasons:
- Common words dominate. "the", "of", "and", "is" appear in literally every chunk. Two completely unrelated chunks share
30% of their words just because English has f
- No word order. "Cats eat fish" and "Fish eat cats" become identical bags. Two paragraphs from different sections that
happen to share vocabulary look identical.

Why n-grams (with N ≈ 5–10) fix both:
- Order-sensitive. An 8-gram like "we propose a novel cross-attention mechanism for" only matches if those 8 words appear
 in that exact order. The probability of an 8ncidence in unrelated text is vanishinglysmall.
- Resilient to local noise. One typo or whitehe ones overlapping that position) — out of~500 in a chunk. The other 492 still match. Substring matching breaks on one character; n-gram overlap degrades
gracefully.
- Cheap to compute. Just set(zip(*[words[i:] for i in range(8)])). Hashing-friendly.

The specific metric we'd use:
overlap = |n-grams(chunk) ∩ n-grams(section)|
Reads as: "what fraction of this chunk's phrases also appear in the gold section?" If 30%+ of the chunk's 8-word phrases
live inside the section, the chunk is overwheat section's content. Mark it gold.

Where else this technique is used: plagiarism engine deduplication, near-duplicate detection in web crawls, the original MinHash/LSH papers. It's the standard fingerprinting tool when you need fuzzy text-match
with order sensitivity.

The 30% threshold is not magic. It's a knob.  section boundaries (which only partiallyoverlap). Too low → unrelated chunks that share boilerplate ("we evaluate on benchmark X") sneak in. 30% is a reasonable
starting default; if your spot-check in task enient or too strict, you'd tune it and re-runthe mapping.

Don't run baseline eval before doing this. Pick 5–10 random rows from eval/benchmark_ragbench.jsonl and manually verify the mapping. Quick check:

python -c "
import json, chromadb
client = chromadb.PersistentClient(path='chroma_db').get_collection('ragbench')
with open('eval/benchmark_ragbench.jsonl') as f:
    for i, line in enumerate(f):
        if i >= 5: break
        row = json.loads(line)
        print('Q:', row['query'])
        print('Gold answer:', row['gold_answer'][:200])
        for cid in row['gold_chunk_ids'][:2]:
            txt = client.get(ids=[cid], include=['documents'])['documents'][0]
            print(f'  [{cid[:20]}…] {txt[:200]}…')
        print()
"

For each row, the chunks shown should plausibly contain the answer to the query. If they don't, the mapping is bad and metrics will be garbage.
